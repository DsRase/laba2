# Лаба 2: классический WAN на IPsec

Сначала выдаём скрипту права и прогоняем его: `chmod +x forge_tunnel.sh && ./forge_tunnel.sh`

Дальше поднимаем стенд: `docker compose up --build -d`

Проверим, что туннель живой и пинг между филиалами идёт: `docker exec branch_a ping -c 3 192.168.77.2`

Теперь смотрим, какой интерфейс смотрит в основной канал (у этой линии ip 10.50.0.11): `docker exec -it branch_a ip -br addr`

Когда нашли интерфейс — глушим его через netem (подставьте свой `<iface>`): `docker exec -it branch_a tc qdisc change dev <iface> root netem loss 100%`

Снова пробуем пинг: `docker exec branch_a ping -c 3 192.168.77.2`. Связи нет — это и есть слабое место статического IPsec: SA жёстко привязана к одному транспортному адресу, резервный канал (10.60.0.0/24) физически на месте, но конфиг про него не знает и ничего автоматически не переключит.

## DoD
1. Готовим конфиги и поднимаем стенд:
   ```
   chmod +x forge_tunnel.sh && ./forge_tunnel.sh
   docker compose up --build -d
   ```
   Ожидаем: оба контейнера `branch_a` и `branch_b` в статусе `Up` (`docker compose ps`).

2. Базовая проверка туннеля, что пинг защищаемого адреса второго филиала:
   ```
   docker exec branch_a ping -c 4 192.168.77.2
   ```
   Ожидаем: `0% packet loss`, RTT около 15–20 мс. Это значит, что IPsec поднялся и трафик идёт через него.

3. Пинг в обратную сторону, чтобы убедиться, что туннель симметричный:
   ```
   docker exec branch_b ping -c 4 192.168.77.1
   ```
   Ожидаем: тоже `0% packet loss`.

4. Находим интерфейс основного канала (тот, у которого адрес из подсети `10.50.0.0/24`):
   ```
   docker exec -it branch_a ip -br addr
   ```
   Запоминаем имя интерфейса (обычно `eth1`).

5. Имитируем обрыв основного канала - ставим 100% потерь:
   ```
   docker exec -it branch_a tc qdisc change dev <iface> root netem loss 100%
   ```

6. Снова пингуем второй филиал:
   ```
   docker exec branch_a ping -c 4 192.168.77.2
   ```
   Ожидаем: `100% packet loss`. Связь не восстанавливается, хотя резервный канал `10.60.0.0/24` физически жив.

7. Возвращаем основной канал в нормальное состояние:
   ```
   docker exec -it branch_a tc qdisc change dev <iface> root netem delay 8ms 1ms
   ```
   Повторный пинг `192.168.77.2` снова даёт `0% packet loss` - туннель сам переподнимается, как только транспорт стал доступен.
